## 1. 体积分析

### 整体大小: **1.4GB**

| 组件 | 大小 | 说明 |
|------|------|------|
| `Electron Framework.framework` | **785 MB** | Chromium + Node.js 运行时 (arm64) |
| `Resources/` (含 app.asar) | **528 MB** | 应用代码、资源文件 |
| `Resources/player/lzc-player.app` | **329 MB** | 独立 Qt 媒体播放器 |
| `core/lzc-core.darwin` | **56 MB** | 后端服务，Go arm64 静态编译 |
| `plugin/cdeplugin` | **18 MB** | 容器插件，Go arm64 |
| `Frameworks/` (其他) | **~3 MB** | Squirrel, Mantle, ReactiveObjC |
| `MacOS/` | **56 KB** | Electron 可执行外壳 |

### app.asar 内部 (96MB packed / 103MB extracted)

应用代码 (`dist/`) 仅 **8.4 MB**，其余为依赖库：

| 包 | 大小 | 说明 |
|---|---|---|
| `dprint-node` | 24 MB | 代码格式化器，开发依赖误打包 |
| `prettier` | 11 MB | 同上 |
| `@babel/*` | 11 MB | Babel 编译工具链，开发依赖误打包 |
| `rxjs` | 9.8 MB | 响应式编程库 |
| `web-streams-polyfill` | 8.6 MB | Web Streams API polyfill |
| `sharp` + 跨平台二进制 | ~6 MB | 图片处理库，含 linux/win 平台文件 |
| 其他依赖 | ~24 MB | 含 protobufjs, axios, vue 等 |
| `dist/` (应用代码) | **8.4 MB** | Vite 打包产物 |

**问题**: 约 46MB 的 devDependencies (`dprint-node`, `prettier`, `@babel`) 被包含在生产环境的 asar 中，属打包配置失误。跨平台 native 模块 (`@img/sharp-linux-x64`, `@img/sharp-win32-x64`) 也未按平台过滤。

### 进程通信模型

```
┌──────────────┐    gRPC-web     ┌──────────────┐
│  Renderer    │ ←────────────── → │  lzc-core    │
│  (Vue SPA)   │   (protobuf)     │  (Go 后端)    │
│              │                  │               │
│  IPC bridge  │                  │  宿主机服务     │
└──────┬───────┘                  └───────────────┘
       │
       │ Electron IPC (contextBridge)
       ▼
┌──────────────┐
│  Main Process│
│  (Node.js)   │
│              │
│  ● gRPC 客户端│
│  ● Koa 服务   │
│  ● 窗口管理   │
│  ● 本地资源   │
└──────────────┘
```

### 通信协议

- **Renderer ↔ Main**: Electron IPC (`ipcRenderer` / `ipcMain`)，通过 `contextBridge` 暴露安全 API
- **Main ↔ lzc-core**: gRPC-web over HTTP，protobuf 序列化
- **Main ↔ lzc-player**: child_process 启动 + stdin/stdout 管道通信
- **Main ↔ 外部**: HTTPS REST (axios)

### 为什么有个独立的 Qt 播放器

Electron 原生 `<video>` 不支持 4K H.265 硬件解码、DTS 音频、MKV 封装、ASS 字幕等。Qt 6.8.3 + ffmpeg 是解决这些问题的成熟方案，代价是 329MB。

## 4. 功能模块

### 用户可见功能

- **账号系统**: 用户名密码、二维码扫码、手机/邮箱验证码、安全码验证
- **设备初始化**: BLE 扫描 → WiFi 配置 → PIN 码验证 → 硬盘加密 → RAID 配置
- **应用商店**: 浏览、安装、卸载、更新，独立窗口运行

- **网络管理**: 有线/无线、WiFi 配置、EAP 认证、网络诊断
- **多用户**: 成员管理、权限控制、受信任设备
- **故障修复**: 重启、回滚、恢复出厂、上传日志
- **系统更新**: 测试版/正式版通道切换
- **远程功能**: DLNA 投屏、远程输入法
- **文件管理**: 文件传输、云盘、安全文件夹
- **Web 终端**: node-pty + WebSocket 终端模拟

### 7.5 Go 二进制中的彩蛋（新增）

#### 1. `"no route to 6.6.6.6"` — 网络连通性探测

**位置**: `lzc-core.darwin` strings
```go
no route to 6.6.6.6
```
用途: 使用 `6.6.6.6`（非标准 IP）检测网络连通性。通常系统探测用 `8.8.8.8` 或 `1.1.1.1`，这里选 `6.6.6.6` 可能是开发者的随机选择或梗（666）。

#### 2. `"BUG: add givisor tun from outside"` — 调试日志

**位置**: `lzc-core.darwin` strings
```
BUG: add givisor tun from outside
```
`"BUG"` 前缀表示开发者将此标记为已知的内部缺陷。"givisor" 疑似 `"divisor"` 或 `"visor"` 的拼写错误。出现在 TUN 设备管理代码中。

#### 3. `"applying darwin workaround..."` — macOS 兼容

**位置**: `lzc-core.darwin` strings
```go
applying darwin workaround...
```
表明 Go 二进制中有针对 macOS 的特定兼容性补丁。macOS 的网络栈与 Linux 差异导致需要 workaround。

#### 4. `"app store 404 bug"` — 应用商店缺陷

**位置**: `lzc-core.darwin` strings 中附近上下文包含 `appstore` 相关错误日志，带有中文标注。
