**SMB Veto 文件列表**（自动隐藏的目录）：
`.media`, `.remotefs`, `.remotefs_old`, `.oneway-sync`, `.lzc_snapshot`, `.otherAppVar`, `.shared-center/`

**compose.override.yml 关键配置：**
- `mountserver`: `privileged: true` — 需要完整系统权限
- `smbforward`: `NET_ADMIN + NET_RAW` — 网络控制权限；`hostname: ${HOSTNAME}`
- `lzcmedia`: 挂载 `/data/snapshot:/lzcapp/run/data/snapshot` 快照卷
- `app`: `LINUX_IMMUTABLE` cap — 可设置文件不可变位；挂载 `/data/snapshot`, `/lzcsys/var`, `/lzcsys/data/document/`（bind with shared propagation）

**权限要求（ext_config）：**
- `PERM_DOCKER_CAP_SYS_ADMIN` — SYS_ADMIN 权限
- `PERM_OTHER_APP_DATA_ADMIN` — 跨应用数据管理
- `PERM_MEDIA_MOUNT_SHARED` — 媒体挂载共享

## 3. 入口脚本与主二进制

### app.sh

```bash
# 清理函数
cleanup() {
  log "应用意外退出!"
  pkill -f lzcinit 2>/dev/null || true
  exit 1
}
trap cleanup EXIT

# 加密配置环境变量
export CLOUD_DRIVE_SECRET_CONFIG_FILE="/lzcapp/pkg/content/config/cloud-drive-secrets.enc.json"
export CLOUD_DRIVE_MASTER_KEY_FILE="/lzcapp/pkg/content/config/cloud-drive-master-key.b64"

exec /lzcapp/pkg/content/backend
```

与相册一致的模式：trap cleanup → pkill lzcinit → exec backend。不同之处在于多了加密配置的环境变量传递。

### WebDAV 协议

从 `backend/dav/usecase` 和 `backend/dav/api` 可推断完整 WebDAV 实现：

- 支持 RFC 4918 WebDAV 协议
- 支持 BasicAuth + Token 认证
- 代理认证处理器（可转发到外部认证）
- 锁定支持（`webdav: unsupported lock info` 表明有但不完全）
- DAV 集合发现（`<D:collection xmlns:D="DAV:"/>`）
- 自定义属性：`X-OC-Mtime`, `X-OC-Ctime`（ownCloud 兼容头）

### SMB 协议

- 标准 SMB/CIFS 协议（端口 445 + 139）
- smbforward 作为前向代理，使用 host 网络模式
- WSDD（Web Services Discovery on Devices）发现协议 (`ENABLE_WSDD=y`)
- SMB 文件 veto 机制（自动隐藏指定目录）
- 文件名限制警告（中文/英文版本）

### 嵌入式 KV 存储 (bbolt)

bbolt 用于以下场景（从函数命名推断）：
- `owsync-ng` 同步任务状态管理

## 8. 加密配置系统

### 架构

```
app.sh 启动时设置:
  CLOUD_DRIVE_SECRET_CONFIG_FILE = /lzcapp/pkg/content/config/cloud-drive-secrets.enc.json
  CLOUD_DRIVE_MASTER_KEY_FILE    = /lzcapp/pkg/content/config/cloud-drive-master-key.b64

cloud-drive-master-key.b64 (Base64 解码 → 32 字节 AES-256 密钥):
  fMP7iIkAeokLFinGSzXQaT2ELK98ZEZqublU+vWxp18=

cloud-drive-secrets.enc.json:
{
  "version": "v1",
  "nonce": "En7LC/MUuetzunTy",           # AES-GCM 12 字节 nonce
  "ciphertext": "SpGcnll4H/..."           # AES-256-GCM 加密密文
}
```

解密流程（`backend/cloud_drive_secretconfig`, 2 funcs）：
1. 读 master key base64 → 解码为 32 字节 AES-256 密钥
2. 读 enc.json → 解析 nonce + ciphertext
3. AES-256-GCM 解密 → JSON 格式的云盘账号配置
4. 解密的配置包含：所有已配置的云盘账号凭据（refreshToken、clientSecret 等）

存储的凭据类型（从字符串推断）：
- Cloud drive credentials（各种云盘的 refreshToken / accessToken）
- OAuth client secrets
- 自动挂载配置
- 高级用户配置

## 12. 服务镜像分析

由于 `registry.lazycat.cloud` 需要 Basic Auth，无法直接拉取分析。以下基于 manifest.yml 和二进制推断。

### lzc-files (主服务镜像)
- 镜像：`registry.lazycat.cloud/lzc/lzc-files:v0.1.55`
- 内容：content.tar 中的 backend + dist + 脚本即为该镜像的文件系统
- base image 推测：`golang:1.24.6-bookworm` + 构建产物
- 动态链接到 glibc，依赖系统库（FFmpeg、p7zip、bbolt 等）

### lzcmedia (媒体处理)
- 镜像：`registry.lazycat.cloud/lzc-media/lzc-media-ng:a2394880`
- 分配 6 个 CPU core
- 环境变量：`MEDIA_ENCODE_VBR=1`, `MEDIA_ENCODE_SET_PARAM=1`
- 功能：视频转码、缩略图生成、媒体元数据提取
- 快照卷挂载：`/data/snapshot:/lzcapp/run/data/snapshot`

### smbserver (SMB 文件共享)
- 镜像：`registry.lazycat.cloud/lzc-files/smbserver:c0ac5dd`
- 基于 samba 协议栈
- 依赖 smbforward
- `DISABLE_AUTOMOUNT_AUTH=y` — 自动挂载无需认证
- 文件名警告定制：`LZC_SMB_WARN_FILENAME_CN/EN`
- Veto 规则过滤敏感目录

### smbforward (SMB 转发)
- 镜像：`registry.lazycat.cloud/lzc-files/forwarder:c0ac5dd`
- `network_mode: host` — 共享宿主机网络
- `ENABLE_WSDD=y` — 启用 WSDD (wsdd2) 网络发现
- `NET_ADMIN + NET_RAW` — 可操作防火墙、原始 socket
- 功能：SMB 前向代理 + 局域网服务发现

### mountserver (文件系统挂载)
- 镜像：`registry.lazycat.cloud/lzc-files/lzc-file-mount:4f3398b`
- `privileged: true` — 容器特权模式（可操作 FUSE、设备挂载）
- 功能：FUSE 挂载远程文件系统

### lzcai (AI 服务)
- 镜像：`registry.lazycat.cloud/ai/lzc-ai:702bd2e1`
- 被 lzcmedia 依赖

### tts (语音合成)
- 镜像：`registry.lazycat.cloud/ai/funasr_cpp:48a508f8`
- 基于 FunASR 的语音识别/合成（阿里达摩院开源项目）
- C++ 实现，性能优先

## 13. 彩蛋与遗留痕迹

### 13.1 拼写错误: `moblie` vs `mobile`

- **位置**: `languages/zh/translation.json` 和 `languages/en/translation.json`
- **内容**: 顶层 key 同时存在 `mobile` 和 `moblie`（拼写错误）
- **分析**: 两个 key 均包含子结构，说明开发者写错了但未清理旧 key。类似相册中的 `Mobil` 错误模式。

### 13.2 空模板文件: template.docx

- **位置**: `content.tar/template.docx`
- **大小**: 0 字节
- **分析**: Word 文档模板为空文件，而 XLSX 和 PPTX 模板正常（6KB 和 33KB）。可能是构建脚本未正确生成，或 LibreOffice 模板生成出错。是遗留的构建产物缺陷。

### 13.3 相册引用: 懒猫相册

- **位置**: `common.json` → `default_picture_name`
- **内容**: zh=`懒猫相册`, en=`Photos`
- **分析**: 网盘的默认图片名称引用相册应用名，暗示两者深度集成。

### 13.4 注释: "简单的应用包装脚本"

- **位置**: `app.sh` 第 3 行
- **内容**: `# 简单的应用包装脚本`
- **分析**: 开发者的自述注释，留着正式产品中未删除。相比相册的 `trap cleanup EXIT` 一行注释都没有，网盘保留了中文注释。

### 13.5 旧版 i18n 备份

- **位置**: `languages/zh/translation_old.json`, `languages/zh/common_old.json`（和 en 对应）
- **分析**: 正式包中包含旧版 i18n 备份文件，说明发布前做了 i18n 更新但忘记清理旧文件。

### 13.6 硬编码云盘域名

- 二进制中硬编码了大量云盘 API 域名、endpoint、client_secret 线索
- 115 网盘: `Ddjpt5B/Cit6EDq2a6cXgxY9lkEIOw4yC1GDF28KrA` — 疑似硬编码的 app key 或 magic 值
- 迅雷: `LXEfS7509300` — 疑似 client ID
- 123 云盘: `512000` — 文件大小限制

### 13.7 LinuxKit 引用

- **位置**: 二进制字符串
- **内容**: `9p`, `v9fs`, `/dev/fuse`, `fuse`, `nfs`, `cifs`
- **分析**: 完整的文件系统协议支持，暗示无盘/容器化部署环境

### 13.8 S3 FIPS 端点列表

- **位置**: 二进制中大量 AWS FIPS endpoint 硬编码
- **分析**: 包含 govcloud、中国区、ISO 区域等数百个 endpoint，远超普通应用需要。可能用于 S3 兼容存储挂载。

### 13.9 GoReSym 模块元数据丢失

- `ModuleMeta` 不可恢复（VA=0, Types=0）
- 这是 stripped Go 1.24 二进制的已知问题 — `.noptrdata` 节可能缺失或损坏
- 意味着类型信息和接口表完全不可恢复

```

https://api.oplist.org/alicloud/renewapi
```
