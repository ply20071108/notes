### 路由映射

| 路径 | 目标 | 说明 |
|---|---|---|
| `/` | `exec://18081` → `lzclightos-lpk` | LightOS 管理后台 |
| `/webshell/` | `exec://8080` → `lcmd-webshell` | 终端 WebShell |
| `/catlink/` | `http://cde:17890` | CDE 容器桌面 WebSocket |
| `/rim/` | `http://cde:8888` | CDE 远程输入法 |
| `/unsafe_api/` | `lzclightos-lpk` 内部路由 | 不安全的发布 API |

### 架构特点

- **OCI-free LPK**: 与懒猫音乐不同，此 LPK 不含 OCI 镜像层；内容直接在 `content.tar` 中
- **双 Go 二进制**: 两个独立的 Go 1.25.4 静态编译的 ELF x86-64 二进制
- **sidecar 容器**: CDE (Container Desktop Environment) 作为独立容器运行
- **SSH 转发守护进程**: 使用 base lzcapp 镜像运行 SSH 转发
- **导出资源**: 导出 `lightos.webshell` 资源，允许其他 LPK 注册为 WebShell Provider

### 2.4 认证架构

```
LightOS 认证流程:
┌──────┐     ┌─────────────┐     ┌──────────────┐
│Client│────▶│lzclightos   │────▶│ LazyCat 系统  │
│      │     │   -lpk      │     │ (gRPC Auth)  │
└──────┘     └─────────────┘     └──────────────┘
    │              │                     │
    │ Cookie       │ Lzc-Auth-Token      │ gRPC GetAuthToken
    │ or Header    │ X-HC-User-Role      │ MasterSecret
    │              │ X-HC-Device-ID      │
```

关键认证变量:
- `LIGHTOS_REQUIRE_COOKIE_AUTH=true` — 需要 Cookie 认证
- `LIGHTOS_LPK_EXTERNAL_PROXY_USER=admin` — 外部代理用户
- `LZCAPP_API_GATEWAY_ADDRESS` — API 网关地址

认证方式:
1. **Cookie 认证** — 检查会话 Cookie
2. **Lzc-Auth-Token Header** — 直接 Token 认证
3. **设备认证** — X-HC-Device-ID + X-HC-User-Role
4. **无 Cookie 回退** — 使用凭据解析

### 3.5 终端工作区模型

```
TerminalWorkspace (工作区)
  ├── Tabs[] (标签页列表)
  │   └── Tab (标签页)
  │       ├── Layout (布局树)
  │       │   ├── SplitNode (分割节点: vertical/horizontal)
  │       │   │   ├── Children[] (子节点)
  │       │   │   └── Size (分割比例)
  │       │   └── Pane (叶子节点 - 终端面板)
  │       │       ├── PTY (伪终端)
  │       │       │   ├── File (PTY master fd)
  │       │       │   └── Process (Shell 进程)
  │       │       ├── OutputBuffer (输出缓冲区)
  │       │       ├── Clients[] (连接的 WebSocket 客户端)
  │       │       └── Activity (活动追踪)
  │       └── ActivePaneID (当前活动面板)
  └── ActiveTabIndex (当前标签页索引)
```

操作:
- `applyAction` — 应用工作区动作
- `createTabLocked` — 创建标签页 (含锁)
- `closeTabLocked` — 关闭标签页
- `splitPaneLocked` — 分割面板 (垂直/水平)
- `closePaneLocked` — 关闭面板
- `movePaneToTabLocked` — 移动面板到其他标签页
- `moveTabLocked` — 移动标签页位置
- `renameTabLocked` — 重命名标签页
- `activateTabLocked` — 激活标签页
- `activatePaneLocked` — 激活面板
- `updateLayoutLocked` — 更新布局

### 3.6 PTY 终端 I/O

```
writePTYInput → PTY master fd → Shell stdin
                    ↓
Shell stdout/stderr → PTY master fd → readLoop → appendOutput
                    ↓
                OutputBuffer (滚动回放)
                    ↓
              attachClient → WebSocket → 浏览器

输入过滤:
  filterGeneratedInputEcho — 过滤自动生成的输入回显
  filterTerminalQueryOutput — 过滤终端查询响应
  filterPrivateControlOutput — 过滤私有控制序列
```

### 3.7 持久化 Agent

Agent 是独立部署的二进制文件，通过 Unix socket 与 lcmd-webshell 通信：

```
Agent 部署流程:
1. ensureAgentBinaryInstalled → 检查/安装 agent 二进制
2. startPersistentAgent → 启动 agent 进程
3. runAgentDaemon → 守护进程循环
4. attachAgentPane → 将 agent 绑定到终端面板

Agent 通信:
  ┌────────────┐     Unix Socket     ┌────────────┐
  │lcmd-webshell│◄──────────────────►│  Agent     │
  │ (主进程)    │   agentRequest/    │ (子进程)   │
  │            │   agentResponse    │            │

```

## 8. 权限与安全

### 8.1 所需权限

```yaml
permissions:
  required:
    - lightos.manage     # 管理 LightOS
    - document.write     # 写入文档
```

### 8.2 安全机制

```
认证层级:
  1. 平台认证 (LazyCat Microserver)
  2. Cookie 认证 (LIGHTOS_REQUIRE_COOKIE_AUTH)
  3. 请求认证 (Lzc-Auth-Token Header)
  4. 设备认证 (X-HC-Device-ID)
  5. 角色验证 (X-HC-User-Role)

安全功能:
  - skip_auth 标志: 允许公开访问 (服务发布时)
  - SSH 转发目标验证 (valid target check)
  - 端口范围限制 (1-65535, 含保留端口检查)
  - 附件路径清理 (sanitizeAttachmentFilename)
  - WebSocket Origin 验证
  - CORS 头写入
```

### 8.3 不安全 API

`/unsafe_api/` 路由提供了未经平台完整认证链的 API 访问：
```
GET    /unsafe_api/services    → 列出服务
POST   /unsafe_api/services    → 创建服务
DELETE /unsafe_api/services    → 删除服务
```

这些 API 被 `public_path` 声明为公开路径：
```yaml
public_path:
  - /webshell/
  - /catlink/
  - /rim/
  - /unsafe_api/
```

## 11. 彩蛋与遗留痕迹

### 11.1 构建版本标记 "dirty"

模块版本为 `v0.0.0-20260616055052-be2fee36697c+dirty` — 表示构建的源码树有未提交的修改。

### 11.2 开发者用户名

构建路径 `gitee.com/linakesi/lightos-admin` 和 `/Users/ponzs/Desktop/os-tree/` 表明：
- Gitee 组织: `linakesi` (林纳斯奇?)
- 开发者本地用户名: `ponzs`
- 项目名: `os-tree` (OS 树)

### 11.3 Ghostty 终端引擎

使用 Ghostty (一个相对新的终端仿真器) 的 WebAssembly 编译版本。Ghostty 本身是使用 Zig 语言开发的，而 WebAssembly 版本通过 Emscripten 编译。这表明团队在追踪前沿的终端技术。

### 11.4 5 种 Nerd Font 预装字体

包含完整的 Nerd Font 字体 - 这是开发者偏好的字体集，包含大量的图标 glyph。特别是 Meslo LG (Powerlevel10k 主题推荐字体) 和 JetBrains Mono (JetBrains IDE 系列推荐字体) 表明开发者社区偏好。

### 11.5 VConsole 式调试面板

调试模式包含 FPS 监视器、性能任务追踪和在线设备面板，这是典型的移动端开发调试工具 (类似微信小程序的 VConsole)。

### 11.6 "PTY 多屏输出说明" 中文化注释

设置中有一段详细的技术说明:
> "PTY 是内核输出，内核无法同时输出不同分辨率的 Shell 结果，所以多设备同时用时，PTY 的输出分辨率会按照正在交互的设备的分辨率输出..."

这段用户可见的说明反映了一个实际工程问题，也表明开发者有向用户解释技术限制的意愿。

### 11.7 前端架构选择: 无框架原生 DOM

终端前端没有使用 Vue/React 等框架，而是使用原生 DOM API 操作。这可能是为了:
1. 最小化 payload 大小 (终端 UI 需要快速加载)
2. 避免框架开销对终端渲染性能的影响
3. 终端的主要 "渲染" 在 canvas 上 (Ghostty 引擎)，DOM 操作较少

### 11.8 `httpPublishTransport.RoundTrip` 实现自定义 HTTP 传输

HTTP 发布系统实现了自定义 `RoundTripper`，这意味着它劫持了 Go 标准库的 HTTP 传输层来做反向代理。

- 发布内部 HTTP 服务到外部网络
- 管理客户端设备

与之前分析的 6 个 LPK 不同，此 LPK 是**纯 Go 编译二进制**，没有 JS/Node.js 后端代码。这标志着 LazyCat 团队对不同场景使用不同技术栈：
| 场景 | 技术栈 | 示例 |
|---|---|---|
| 简单 CRUD 应用 | Node.js + Express | 懒猫音乐 |
| 系统管理/网络组件 | Go (编译) | LightOS, lcmd-webshell |
| 前端终端 UI | 原生 JS + Ghostty WASM | WebShell |
| 桌面环境 | Alpine Linux sidecar | CDE Runtime |

### 关键发现

1. **双二进制架构**: lzclightos-lpk (23MB) + lcmd-webshell (22MB)，共享 lzc-sdk 依赖
2. **60+ Go 源码文件**: 功能覆盖访问控制、SSH 转发、HTTP 发布、WebShell Provider、构建管理等
3. **完整的 SSH 转发系统**: 端口分配/绑定/探测/持久化，支持 Yamux 多路复用
4. **HTTP 发布系统**: 可将内部容器服务发布到外部，含动态 LPK 生成
5. **CDE/Catlink 集成**: 完整的桌面环境远程控制栈
6. **30+ 终端主题**: 包含所有流行配色方案
7. **Ghostty Web 终端**: 前沿的 Zig→WASM 编译终端引擎
8. **不安全 API 暴露**: `/unsafe_api/` 绕过完整认证链
